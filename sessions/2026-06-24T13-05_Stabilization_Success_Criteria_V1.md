# 📈 시스템 안정화 테스트 성공/실패 기준 정의서 (V1)

## 🎯 목표: 핵심 가치 증명 시스템($SSI$)의 객관적 데이터 기반 마련
본 문서는 KPI Acceptance Criteria에 근거하여, 비즈니스 의사결정 과정에서 필수적인 세 가지 핵심 지표 ($\Delta R, P_{stab}, T_{recovery}$)의 **테스트 성공/실패 기준(Acceptance Criteria)**을 명확히 정의하는 것을 목표로 합니다.

---
### 1. $\Delta R$: 리스크 감소율 (Risk Reduction Rate) - 비즈니스 관점
**[정의]**: 시스템적 불안정성으로 인해 고객이 인지할 수 있는 포트폴리오/자산의 잠재적 손실 규모가 얼마나 줄었는지를 정량화하는 지표.
**[KPI 연결고리]**: 사용자가 '만약 시스템에 문제가 없다면' 발생했을 리스크(Baseline Risk)와, 우리가 제공하는 서비스로 인해 예측되는 실제 리스크 간의 격차를 측정합니다.

| 항목 | 성공 기준 (Success Criteria) | 실패 기준 (Failure Criteria) |
| :--- | :--- | :--- |
| **측정 단위** | % (퍼센트) | N/A |
| **수치 요구사항** | 테스트 기간 동안, 평균 $\Delta R$이 최소 10% 이상 지속적으로 관찰되어야 함. | 임의의 시점 또는 특정 조건(예: 시장 급변동)에서 $\Delta R$ 수치가 5% 미만으로 하락하는 현상이 발생함. |
| **데이터 요구사항** | 백테스팅 데이터 및 실시간 시뮬레이션 데이터가 최소 3가지 이상의 다양한 시장 환경(상승, 하락, 횡보)을 커버해야 함. | 특정 패턴의 데이터만 반복적으로 수집되거나, 충분한 기간 동안 데이터 연속성이 끊김. |

### 2. $P_{stab}$: 시스템 안정성 지표 (Platform Stability Index) - 기술/운영 관점
**[정의]**: 핵심 기능(SSI 보고서 생성, API 호출 등)이 중단 없이 작동할 수 있는 기술적 신뢰도를 측정합니다. 이는 고객에게 '절대적인 믿음'을 제공하는 기반입니다.
**[KPI 연결고리]**: 시스템 다운타임(Downtime)의 직접적인 역수 개념으로 활용됩니다.

| 항목 | 성공 기준 (Success Criteria) | 실패 기준 (Failure Criteria) |
| :--- | :--- | :--- |
| **측정 단위** | % (가용성, Availability) | N/A |
| **수치 요구사항** | 24시간 운영 시간 중 가동률(Uptime)이 최소 99.9%를 달성해야 하며, 장애 발생 시 자동 복구 시간이 10분 이내여야 함 (Self-Healing 루프 검증). | 특정 API 호출에서 빈번하게 `HTTP 5xx` 에러 코드가 반환되거나, 수동 개입이 필요한 다운타임이 발생하는 경우. |
| **데이터 요구사항** | 모든 핵심 마이크로서비스(MSA)의 로그와 상태 코드(Health Check)가 실시간으로 모니터링되고 기록되어야 함. | 특정 시간대에 데이터 소스의 연결 오류 또는 네트워크 지연 시간이 3초를 초과하는 현상이 발생함. |

### 3. $T_{recovery}$: 복구 시간 (Time to Recovery) - 대응/회복 관점
**[정의]**: 시스템에 치명적인 장애(Critical Failure)가 발생했을 때, 비즈니스 프로세스를 재개하기까지 걸리는 최소한의 시간을 측정합니다. 이는 고객에게 '대비 능력'을 증명하는 지표입니다.
**[KPI 연결고리]**: 서비스 중단으로 인한 잠재적 매출 손실액($L$)과 연관하여 계산됩니다.

| 항목 | 성공 기준 (Success Criteria) | 실패 기준 (Failure Criteria) |
| :--- | :--- | :--- |
| **측정 단위** | 분(Minute) 또는 초(Second) | N/A |
| **수치 요구사항** | 시나리오 기반 장애 발생 후, 백업 시스템 전환을 통해 핵심 기능을 정상화하는 데 걸리는 시간이 30분 이내여야 합니다. (최대 허용 시간: $T_{max}$). | 재난 복구 계획(DRP)에 정의된 매뉴얼대로 진행했음에도 불구하고, 특정 모듈의 데이터 정합성을 확보하지 못해 수동 검토가 필요한 경우. |
| **데이터 요구사항** | 백업 시스템 전환 프로세스가 자동화 스크립트로 완전히 구현되어야 하며, 그 실행 과정과 로그가 기록되어야 함. | 장애 복구 시도 자체가 실패하거나, 이전 데이터를 활용하기 위해 데이터 정제(Data Cleansing) 작업이 수동으로 필요함. |
